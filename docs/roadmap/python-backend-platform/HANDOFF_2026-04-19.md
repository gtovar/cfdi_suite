# Handoff 2026-04-19

## Qué quedó hecho

- se reconciliaron `STATUS.md` e `IMPLEMENTATION_PLAN.md` contra la implementación real
- quedó documentado que el provider contract interno rico ya existe en `backend/app/providers/base.py`
- quedó documentado que `run_analyze_cfdi` ya publica semántica v1 estable para `issues` y `meta`
- quedó documentado que el frontend todavía ejecuta el fallback TS, pero solo ante fallas técnicas elegibles
- quedó fijado como siguiente frente inmediato: contract tests HTTP v1 + fallback governance en backend

## Estado real al cierre

- el backend ya no está en fase "mínima" o "por crear"
- el contrato HTTP v1 ya existe y ya tiene shape/semántica básicas cerradas
- `cfdi.findings` sigue siendo la fuente oficial, pero todavía se entrega vacío desde Python
- el fallback local TS sigue vivo como transición
- la policy de fallback todavía no pertenece completamente a la plataforma porque `src/lib/cfdi-api-client.ts` aún decide cuándo activar fallback por falla técnica

## Siguiente paso mínimo recomendado

Sacar la policy de fallback del frontend sin romper el contrato HTTP v1:

1. introducir un orquestador backend `primary + fallback`
2. usar `ProviderIssue` + `ProviderDiagnostics.fallback_eligible` para decidir elegibilidad
3. publicar el resultado alterno usando el mismo contrato v1
4. marcar `meta.providerMode = "fallback"` y `meta.fallbackReason`
5. dejar el frontend sin policy de fallback por status HTTP; como transición, permitir solo escape hatch por API totalmente indisponible si todavía hace falta

## Prioridad explícita para la siguiente sesión

1. agregar contract tests del endpoint `POST /api/cfdi/analyze`
2. mover fallback governance al backend
3. no implementar findings ricos todavía
