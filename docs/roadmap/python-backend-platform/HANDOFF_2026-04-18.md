# Handoff 2026-04-18

## Qué quedó hecho

- backend FastAPI creado en `backend/app/`
- endpoint `GET /api/health`
- endpoint `POST /api/cfdi/analyze`
- provider `python-satcfdi` conectado por wrapper Python existente
- interfaz base de provider creada para futuros hooks
- cliente HTTP del frontend creado
- `useCfdiAnalysis` cambiado para usar API como camino principal
- fallback TypeScript conservado
- proxy de Vite configurado hacia `/api`
- script `npm run dev:api` agregado
- README actualizado con flujo de desarrollo frontend + API

## Qué falta para usarlo de verdad

- instalar dependencias Python del backend:

```bash
pip install -r backend/requirements.txt
```

- arrancar API:

```bash
npm run dev:api
```

- arrancar frontend:

```bash
npm run dev
```

## Limitaciones conocidas

- el backend actual devuelve estructura CFDI y filas, pero no findings equivalentes al motor TS
- `cfdi.findings` ya es la fuente oficial, pero hoy sigue llegando vacío desde el backend Python
- el fallback TS sigue siendo importante para comparar comportamiento
- el contrato implementado ya refleja la semántica v1 mínima de `requestId`, `providerMode`, `degraded` y fuente oficial única de findings

## Siguiente frente recomendado al retomar planeación

No seguir primero por validación manual del backend mínimo.

El siguiente frente correcto es cerrar y usar como referencia única:

- ownership operativo de `analyze_cfdi`
- semántica oficial del contrato HTTP v1
- mínimo operativo v1 para fallback, trazabilidad y datos sensibles

La validación manual con fixtures sigue siendo útil, pero debe ocurrir después de fijar esa línea base contractual y operativa para no validar contra un contrato todavía ambiguo.
